## 1. Cách Dùng Tối Thiểu

```python
class Product(TRCFBaseModule):
    _name        = "products"
    _description = "Sản phẩm"

    name  = fields.CharField(required=True, label="Tên món")
    price = fields.DecimalField(precision=12, scale=2, label="Giá bán", ui_type="currency")
```

→ Tự sinh: DB table, Pydantic schemas, async CRUD, **9 REST endpoints** + `/meta/schema`.

> **Thiết kế:** TRCFBaseModule là **CRUD base class** cho sử dụng nội bộ. Mỗi module là 1 class độc lập, không thiết kế để cross-module inheritance (kiểu Odoo `_inherit`). Muốn thêm field → sửa trực tiếp class. Muốn chia sẻ logic → dùng Python mixin hoặc class inheritance thông thường.

> **ForeignKeyField display convention:** Khi frontend render `<Select>` cho ForeignKey, mặc định dùng `name` field của related module làm label. Nếu related module không có `name`, khai báo `display_field`.
> Ví dụ: `user_id = fields.ForeignKeyField(to="users", display_field="email", label="Người dùng")`

## 5. Archive (Lưu Trữ) — Giống Odoo `active` Field

Mặc định `_archive = True`. Khi `POST /products/1/archive`:
- `active = False` (lưu trữ, không xóa khỏi DB)
- Mọi query tự lọc `WHERE active = TRUE`
- `GET /products/1` → 404 nếu đã lưu trữ
- `POST /products/1/restore` → khôi phục (`active = True`)
- `DELETE /products/1` → xóa vật lý (cần permission `delete`)
- `GET /products/?with_archived=true` → bao gồm cả đã lưu trữ (chỉ khi đã đăng nhập)

> **Lưu ý:** `with_archived=true` chỉ hoạt động khi user đã đăng nhập (có JWT hợp lệ).
> Nếu `_require_auth=False` và request không có token → `with_archived` bị bỏ qua tự động.

Tắt archive:
```python
class AuditLog(TRCFBaseModule):
    _archive = False  # xóa vật lý hoàn toàn, không lưu trữ
```

> **Archive + `unique=True`:** Khi `_archive=True`, metaclass tự sinh **partial unique index** cho PostgreSQL:
> ```sql
> CREATE UNIQUE INDEX ix_products_code_active ON products (code) WHERE active = TRUE;
> ```
> → Record lưu trữ (`active=False`) **KHÔNG chiếm unique** → tạo mới cùng mã không bị lỗi.
> Đây là cách Odoo xử lý: archive sản phẩm cũ → tạo mới cùng mã → hoạt động bình thường.

## 6. Relationship

### 6a. ForeignKey (Một-Nhiều)

```python
class Product(TRCFBaseModule):
    category_id = fields.ForeignKeyField(to="categories", label="Danh mục")
```

> **`to` là gì?** Giá trị `_name` của module đích, hoặc tên bảng CMS Core (`users`, `groups`).
> Nếu `to` không tìm thấy → raise error khi app startup.

**Behavior:**
- Validate `category_id` tồn tại khi create/update → `422` nếu sai
- GET/LIST tự JOIN → trả nested object:
```json
{"id": 1, "name": "Cà phê sữa", "category_id": 2,
 "category": {"id": 2, "name": "Coffee"}}
```
- Xóa category còn product → `400 Bad Request` (tự detect FK)
- `on_delete="CASCADE"` → xóa cascade
- `on_delete="SET NULL"` (default) → set FK = NULL khi xóa related
- `on_delete="RESTRICT"` → chặn xóa nếu còn FK reference

### 6b. ManyToMany (Nhiều-Nhiều)

```python
class Product(TRCFBaseModule):
    tags = fields.ManyToManyField(to="tags", label="Tags")
    # Tự tạo bảng trung gian: product_tags (product_id, tag_id)
```

**Behavior:**
- POST/PUT body: `{"name": "Cà phê", "tags": [1, 2, 3]}` (luôn dùng IDs)
- **LIST response:** `{"name": "Cà phê", "tags": [1, 2, 3]}` (IDs — nhẹ, nhanh)
- **GET single response:** `{"name": "Cà phê", "tags": [{"id": 1, "name": "Hot"}, ...]}` (nested objects — đủ data render)
- Cập nhật tags: gửi list id mới → tự sync bảng trung gian

### Handler Pipeline — Thứ Tự Xử Lý Bên Trong Mỗi Endpoint

> Dev cần biết rõ thứ tự để implement đúng. Mỗi bước lỗi → dừng ngay, trả error.

**CREATE (`POST /{name}/`):**
```
1. Parse body → dict
2. AUTH:       _require_auth + _public_actions → action nằm trong _public_actions thì bỏ qua auth
               → không public + user_id is None → 401
3. PERMISSION: query group_permissions → không có quyền "create" → 403
4. AUDIT:      data["created_by"] = user_id          ← handler layer, tự động
5. READONLY:   strip _readonly_fields khỏi data
6. FK:         kiểm tra ForeignKey tồn tại trong DB  → 422 nếu sai
7. COMPUTED:   tính stored computed fields
8. DB:         INSERT + commit
9. RESPONSE:   {success: true, data: {...}, message: null}
```

**UPDATE (`PUT /{name}/{id}`):**
```
1. Parse body (exclude_unset) → dict
2. AUTH + PERMISSION ("update")
3. FETCH:      lấy record hiện tại → 404 nếu không tồn tại hoặc active=False
4. VERSION:    kiểm tra optimistic lock (nếu _optimistic_lock=True) → 409
5. AUDIT:      data["updated_by"] = user_id
6. READONLY:   strip _readonly_fields
7. FK:         validate ForeignKey
8. COMPUTED:   recompute stored computed fields (nếu depends thay đổi)
9. DB:         UPDATE + commit
10. RESPONSE
```

**ARCHIVE (`POST /{name}/{id}/archive`):**
```
1. AUTH + PERMISSION ("archive")
2. FETCH:      lấy record → 404 nếu không tồn tại hoặc active=False
3. DB:         set active=False (lưu trữ)
4. RESPONSE
```

**DELETE (`DELETE /{name}/{id}`):**
```
1. AUTH + PERMISSION ("delete")
2. FETCH:      lấy record → 404 nếu không tồn tại
3. DB:         xóa vật lý khỏi DB
4. RESPONSE
```

**LIST (`GET /{name}/`):**
```
1. AUTH + PERMISSION ("list")
2. BASE QUERY: SELECT ... WHERE active = TRUE
3. SEARCH:     thêm ILIKE trên _search_fields
4. FILTER:     parse filter prefix syntax trên _filter_fields
5. SORT:       sort_by + sort_desc
6. PAGINATE:   skip + limit (cap bởi _max_page_size)
7. FK JOIN:    auto JOIN related tables
8. COMPUTED:   tính non-stored computed fields cho mỗi record
9. RESPONSE:   {success: true, data: {total, items, skip, limit}}
```

**GET (`GET /{name}/{id}`):**
```
1. AUTH + PERMISSION ("read")
2. FETCH:      lấy record → 404 nếu không tồn tại hoặc active=False
3. FK JOIN:    auto JOIN related tables
4. COMPUTED:   tính non-stored computed fields
5. RESPONSE
```

## 8. Filter Prefix Syntax

| Prefix | SQL | Ví dụ |
|---|---|---|
| *(không có)* | `=` | `?status=active` |
| `[>]` / `[>=]` | `>` / `>=` | `?price=[>=]25000` |
| `[<]` / `[<=]` | `<` / `<=` | `?stock=[<]10` |
| `[!=]` | `!=` | `?status=[!=]cancelled` |
| `[IN]` | `IN` | `?category_id=[IN]1,2,3` |
| `[NOT IN]` | `NOT IN` | `?status=[NOT IN]draft,cancelled` |
| `[LIKE]` | `ILIKE %val%` | `?name=[LIKE]coffee` |
| `[BETWEEN]` | `BETWEEN` | `?price=[BETWEEN]20000,50000` |

> **Auto-columns luôn filterable:** Các cột tự thêm (`created_at`, `updated_at`, `created_by`, `updated_by`, `active`) luôn có thể filter mà **không cần khai báo** trong `_filter_fields`.
> Ví dụ: `?created_at=[BETWEEN]2024-01-01,2024-12-31` hoặc `?created_by=5` luôn hoạt động.

> ⚠️ **Bảo mật:** Tất cả filter values **PHẢI** dùng **parameterized queries** (SQLAlchemy `column.operator(value)`).
> **KHÔNG BAO GIỞ** string concat SQL. SQLAlchemy ORM tự handle nếu dùng đúng `.where()` + column operators.
> Ví dụ đúng: `query.where(Product.price >= Decimal(value))`
> Ví dụ sai: `query.where(text(f"price >= {value}"))` ← SQL injection!

## 10. Audit Trail (Tự Động)

`created_by` và `updated_by` được inject ở **handler layer** tự động:

```python
# base/module.py — handler tự inject:
async def _handle_create(request, body, db, module_cls):
    data = body.dict()

    # 1. Inject audit trail (handler layer — tự động, dev KHÔNG can thiệp)
    data["created_by"] = getattr(request.state, "user_id", None)

    # 2. Strip readonly fields
    # 3. Validate FK
    # 4. Insert DB
    ...

async def _handle_update(request, body, db, module_cls, id):
    data = body.dict(exclude_unset=True)

    # 1. Inject audit trail (handler layer)
    data["updated_by"] = getattr(request.state, "user_id", None)

    # 2. Strip readonly, validate FK, update DB
    ...
```

> **Tại sao inject ở handler?**
> Đảm bảo audit trail **luôn có**, bất kể module nào kế thừa.
> Đây là cách Odoo xử lý `create_uid` / `write_uid`.

## 12. Computed Fields

> Dùng `ComputedField` dataclass (từ `base/fields.py`). Không phải `BaseField` — không tạo DB column trực tiếp.

### 12a. Non-stored Computed (tính khi trả response)

Tính khi trả response, **không lưu DB**:

```python
from base.fields import ComputedField

class OrderItem(TRCFBaseModule):
    quantity   = fields.IntField(required=True, label="Số lượng")
    unit_price = fields.DecimalField(precision=12, scale=2, required=True, label="Đơn giá", ui_type="currency")

    # Non-stored computed (default: store=False)
    subtotal = ComputedField(
        fn=lambda r: r.quantity * r.unit_price,
        depends=["quantity", "unit_price"],
        label="Thành tiền",
    )
    # Response: {..., "subtotal": 58000.0}
```

### 12b. Stored Computed Fields (lưu DB)

> Computed field **lưu vào DB** → có thể filter, sort, index.
> Tự recompute khi dependency fields thay đổi.
> Cảm hứng: Odoo `compute + store=True` + `@api.depends`.

```python
class OrderItem(TRCFBaseModule):
    quantity   = fields.IntField(required=True, label="Số lượng")
    unit_price = fields.DecimalField(precision=12, scale=2, required=True, label="Đơn giá")

    # Stored computed field:
    subtotal = ComputedField(
        fn=lambda r: r.quantity * r.unit_price,
        depends=["quantity", "unit_price"],
        store=True,                      # lưu DB → cho phép filter/sort
        label="Thành tiền",
    )
```

### ComputedField Dataclass

```python
# base/fields.py
@dataclass
class ComputedField:
    fn: Callable[[record], Any] = None    # Hàm tính: nhận record, trả value
    depends: list[str] = []               # Fields phụ thuộc → trigger recompute
    store: bool = False                   # True = lưu DB, False = tính khi response
    label: str = ""                       # Label cho frontend
    _name: str = ""                       # Auto-set bởi __init_subclass__
```

**Hành vi:**
- `POST /order-items/` → nếu `store=True`: compute + lưu `subtotal` vào DB
- `PUT /order-items/1` với `{"quantity": 3}` → tự recompute `subtotal` (nếu dependency thay đổi)
- `store=True`: `?sort_by=subtotal` và `?subtotal=[>=]100000` hoạt động
- `store=False`: chỉ tính khi trả response, không filter/sort được
- Dev **không gửi** `subtotal` trong body → bỏ qua (computed field không cho set thủ công)
- `/meta/schema` response chứa `computed_fields` riêng (không nằm trong `fields`)

> **Serialization:** Decimal → `float`, datetime → ISO string. Handler tự convert.

## 13. Custom Validation

Validate data dùng **Pydantic field validators** (tự sinh trong schema) hoặc **DB constraints**:

```python
class Product(TRCFBaseModule):
    _name = "products"

    price = fields.DecimalField(
        precision=12, scale=2,
        required=True,
        label="Giá bán",
        min_value=0,  # Tự sinh Pydantic validator: giá >= 0
```

## 14. Bulk Create / Update (JSON)

```
POST /{name}/bulk
Content-Type: application/json
Body: [{...}, {...}, ...]
Query: ?upsert=true|false (default: false)
```

**Hành vi:**
- Nhận **JSON array** của objects
- Validate SelectionField + FK cho từng row
- Nếu `upsert=false` (default): chỉ create mới
- Nếu `upsert=true`: row có `id` tồn tại → UPDATE, không có → CREATE
- Transaction atomic: lỗi → rollback toàn bộ
- Trả về chuẩn format:
```json
{"success": true, "data": {"created": 95, "updated": 3, "errors": [{"row": 3, "detail": "..."}]}, "message": "Import thành công: 95 tạo mới, 3 cập nhật"}
```

### Permission Check Flow (TRCFBaseModule dùng CMS Core data)

```
Request đến TRCFBaseModule endpoint (ví dụ: POST /products/1/archive)
    ↓
_require_auth = True?
    request.state.user_id is None → 401 Unauthorized
    ↓
request.state.user_is_superuser = True?
    → Cho qua toàn bộ, không check gì nữa
    ↓
TRCFBaseModule query bảng group_permissions (CMS Core table):
    WHERE module_name = "products"
      AND action = "archive"

    → Không tìm thấy → 403 Forbidden
```

## 23. Thiết Kế: Không Dùng Cross-Module Inheritance

> **TRCFBaseModule KHÔNG thiết kế để cross-module inheritance** (kiểu Odoo `_inherit`).
> Mỗi module là 1 class độc lập, khai báo đầy đủ fields riêng.

**Lý do:**
- Cross-module inheritance sinh ra nhiều vấn đề phức tạp: field collision, hook chain, M2M naming...
- Hệ thống này dùng **nội bộ**, không cần mở rộng kiểu open-source ERP
- Muốn thêm field vào module → **sửa trực tiếp class**

**Cách chia sẻ logic giữa modules:**

```python
# Dùng Python mixin thông thường:
class TimestampMixin:
    """Mixin dùng chung cho các module cần start_date/end_date"""
    start_date = fields.DateTimeField(label="Bắt đầu")
    end_date   = fields.DateTimeField(label="Kết thúc")

class Promotion(TimestampMixin, TRCFBaseModule):
    _name = "promotions"
    name  = fields.CharField(required=True, label="Tên khuyến mãi")
    # Có cả start_date, end_date từ mixin

class Event(TimestampMixin, TRCFBaseModule):
    _name = "events"
    title = fields.CharField(required=True, label="Tiêu đề")
    # Có cả start_date, end_date từ mixin
```

> **Nguyên tắc:** Dùng **Python class inheritance + mixin** cho logic dùng chung. Không cần cơ chế `_inherit` của Odoo.

### Hành Vi

4. **Force re-seed:** admin có thể xóa row trong `_module_seeds` → lần khởi động sau sẽ seed lại (chỉ khi bảng trống)

```python
        if not already_seeded:
            count = await db.scalar(select(func.count(cls._model.id)))
            if count == 0:
                data = load_json(cls._seed_data)
                for row in data:
                    db.add(cls._model(**row))
                db.add(_ModuleSeed(module_name=name))
                await db.commit()
```

> **Dev chỉ cần:** tạo file JSON trong `data/` + khai báo `_seed_data` → app tự seed khi khởi động lần đầu.

### SystemSetting Model (CMS Core)

```python
# CMS Core — cố định, viết tay
class SystemSetting(Base):
    __tablename__  = "system_settings"
    __table_args__ = (
        UniqueConstraint("module_name", "key"),  # chống conflict: mỗi module chỉ có 1 key duy nhất
    )

    id          = Column(Integer, primary_key=True)
    module_name = Column(String(255), nullable=False, index=True)  # "attendances", "products"
    key         = Column(String(255), nullable=False)               # "late_threshold_min"
    value       = Column(Text, nullable=False)                      # "15" (luôn lưu string, cast khi đọc)
    value_type  = Column(String(50), nullable=False)                # "string"|"integer"|"decimal"|"boolean"|"json"

    updated_at  = Column(DateTime(timezone=True), onupdate=func.now())
```

### Seed Startup (Chống Conflict)

```python
# main.py — sau khi đăng ký modules:
for cls in [Category, Product, Order, Attendance]:  # danh sách tường minh
    name = cls._name
    for setting in cls._settings:
        # INSERT ... ON CONFLICT (module_name, key) DO NOTHING
        # → Chỉ insert nếu chưa có. KHÔNG ghi đè value superuser đã sửa.
        existing = await db.scalar(
            select(SystemSetting).where(
                SystemSetting.module_name == name,
                SystemSetting.key == setting["key"],
            )
        )
        if not existing:
            db.add(SystemSetting(
                module_name=name,
                key=setting["key"],
                value=str(setting["default"]),
                value_type=setting["type"],
                label=setting["label"],
            ))
    await db.commit()
```

> **An toàn:** Nếu superuser đã sửa `late_threshold_min` từ 15 → 30, app restart sẽ **không ghi đè** về 15.
> Chỉ insert settings mới (chưa có trong DB). Settings đã tồn tại → giữ nguyên.

### Helper Function Đọc Setting

```python
# auth/utils.py hoặc base/utils.py
import json
from decimal import Decimal

async def get_setting(module: str, key: str, db: AsyncSession):
    """Lấy setting từ DB, auto cast theo value_type."""
    row = await db.scalar(
        select(SystemSetting).where(
            SystemSetting.module_name == module,
            SystemSetting.key == key,
        )
    )
    if not row:
        return None
    match row.value_type:
        case "integer": return int(row.value)
        case "decimal": return Decimal(row.value)
        case "boolean": return row.value.lower() in ("true", "1", "yes")
        case "json":    return json.loads(row.value)
        case _:         return row.value  # string

# Sử dụng:
threshold = await get_setting("attendances", "late_threshold_min", db)  # → 15 (int)
```

## 27. CLI Management Tool (manage.py)

Hệ thống cung cấp một công cụ dòng lệnh (`backend/app/manage.py`) để quản lý database và dữ liệu mẫu (Seed data) ở mức độ granular (theo từng module).

### 27a. Mục Tiêu & Cơ Chế
Dùng để đẩy nhanh quá trình phát triển (Development) khi cấu trúc module thay đổi liên tục. Thay vì dùng `Alembic migration` phức tạp trong giai đoạn dev, ta có thể reset nhanh một module đơn lẻ.

### 27b. Các Lệnh Hỗ Trợ

| Lệnh | Ví dụ | Mô tả |
|---|---|---|
| `list-modules` | `python backend/app/manage.py list-modules` | Liệt kê tất cả các module đang có trong `REGISTERED_MODULES` |
| `reset-module` | `python backend/app/manage.py reset-module products` | Reset duy nhất một module (Drop -> Create -> Seed) |
| `reset-all` | `python backend/app/manage.py reset-all` | Xóa sạch và tạo lại toàn bộ database |

### 27c. Quy Trình `reset-module`
Khi lệnh `reset-module <name>` được gọi, script thực hiện các bước:
1.  **Drop Table**: Xóa bảng chính của module (VD: `products`).
2.  **Drop Junctions**: Xóa tất cả các bảng trung gian liên quan (VD: `products_units` cho Many-to-Many).
3.  **Create All**: Chạy lệnh `create_all` của SQLAlchemy (chỉ tạo các bảng còn thiếu).
4.  **Seed Data**: Đọc file JSON tương ứng tại `backend/seeds/<name>.json` và nạp vào DB.

### 27d. Seed Data Folder System
Hệ thống tự động tìm dữ liệu mẫu dựa trên `_module_name` của Class:
- **Đường dẫn**: `backend/seeds/`
- **Định dạng**: JSON array của các objects.
- **Tên file**: Phải đặt tên trùng với `_module_name` (Ví dụ: `products.json`, `categories.json`).

> **Lưu ý:** Nếu một module có ForeignKey bắt buộc (vd: `unit_id`), cần đảm bảo module liên quan đã được Seed trước hoặc đã có sẵn dữ liệu trong DB.
